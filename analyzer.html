Analizador de ensamblador x86

Pega instrucciones (sintaxis Intel, una por linea). Para cada una se busca su forma en la base de datos y se reporta lo que hace y su coste en la microarquitectura elegida.
Como funciona el analisis

Para cada instruccion (sintaxis Intel, una por linea; se ignoran etiquetas nombre: y comentarios ; # //) se identifica el mnemonico, se clasifica cada operando (registro y su ancho, memoria — el ancho se toma del prefijo qword ptr [rax] o del otro operando —, inmediato) y se localiza su forma en la base de datos.

Del bloque completo se calculan las dos cotas que gobiernan el tiempo de ejecucion en la microarquitectura elegida:

front-end
Σ uops ÷ el ancho de decodificacion/emision de la microarquitectura: los ciclos si limitara el reparto de µops al motor de ejecucion (superescalar).
throughput
Σ del throughput reciproco de cada instruccion: los ciclos si limitaran los puertos de ejecucion (instrucciones independientes, encauzadas en paralelo).
latencia (camino critico)
se construye el grafo de dependencias reales entre instrucciones (registros — con sus alias de ancho —, flags y memoria) y se recorre el camino mas largo: los ciclos si limitaran las cadenas de dependencia.

La estimacion del bloque es el mayor de las tres, y se indica el cuello de botella. La eliminacion de movimientos (mov registro-registro a 0 ciclos) se tiene en cuenta.

Debajo, el codigo optimizado: aplica peephole (puesta a cero mov r,0→xor r,r, reduccion de fuerza imul r,2^k→shl r,k, copia a si mismo), eliminacion de codigo muerto y reordenacion valida (respeta todas las dependencias) para favorecer la ejecucion paralela; muestra que aplico, por que, y el mismo analisis para comparar.
